### 🧠 배운 점 
- **[자바 프로젝트 생성 과정]**
  - GitHub에서 저장소를 만든 뒤 IDE에서 열면 자동으로 clone되어 로컬 작업이 가능하다.
  - GitHub 주소를 clone하면 IDE를 바꿔도 동일한 프로젝트를 사용할 수 있다. (VSCode &rightarrow; IntelliJ)
  - Gradle 등 빌드 도구는 기본적으로 `src/main/java` 구조를 사용하며, 이 구조를 따르면 IDE에서 자바 프로젝트로 자동 인식된다.
  - `Sources Root`에 등록되어 있어야 IntelliJ가 자바 소스 코드 폴더로 인식한다.

- **[데이터 구조 JSON vs CSV]**
  - **JSON:** 계층적 구조로, 중첩된 객체·리스트 표현이 가능하며 복잡한 데이터(예: 노래 + 아티스트 + 태그)를 다루기에 적합하다.
  - **CSV:** 단순한 행·열 형태의 2차원 표 구조로, 숫자나 문자열 기반의 목록 데이터를 저장하거나 분석할 때 효율적이다.

### 💬 회고 
- **자바 프로젝트 설정에 하루 대부분을 사용했다.**
  - GitHub에서 먼저 저장소를 만든 뒤 clone했더니,
Sources Root, Gradle, 패키지 폴더 등을 전부 수동으로 설정해야 했다.
  - 다음부터는 **IntelliJ에서 프로젝트를 먼저 생성하고 → 이후 GitHub에 push**하는 방식이 훨씬 효율적일 것 같다.

- **`Sources Root` 설정 실수**
  - 패키지명이 계속 `package findmyvibe;`가 아닌 `package main.java.findmyvibe;`처럼 전체 경로가 다 떠서 그 원인을 한참 찾았다.
  - 원인은 java 패키지만 Sources Root로 지정했어야 했는데 src 전체를 지정해버린 것이었다. 
  - 이렇게 하면 src 밑의 모든 폴더(main/java/...)가 전부 패키지명에 포함된다고 인식된다.
  - Sources Root는 **자바 코드의 시작점**이자 **package 선언의 기준점**이므로, **실제 패키지명이 시작되는 폴더(java)를 Source Root로 지정해야** 한다.

### 🧠 배운 점
- **[스프링의 핵심]**
  - **IoC(Inversion of Control, 제어의 역전)**: 객체의 생성/관리를 개발자가 아닌 프레임워크가 대신 함
  - **DI(Dependency Injection, 의존성 주입)**: 외부에서 객체를 주입받아 사용하는 것
  
    &rightarrow; 외부(스프링 컨테이너)에서 생성/관리하는 객체를 클래스 안으로 가져와서 사용 (DI가 가능함으로써 IoC를 할 수 있다)
    
    &rightarrow; 스프링 컨테이너에서 관리하는 객체를 '빈'이라고 함. 빈은 애너테이션을 통해 주입받을 수 있음.

  - **AOP(Aspect Oriented Programming, 관점 지향 프로그래밍)**: 프로그래밍을 할 때 핵심 관점과 부가 관점을 나누어서 개발하는 것
  - **PSA(Portable Service Abstraction, 추상화)**: 어느 기술을 사용하던 일관된 방식으로 처리하도록 하는 것

### 💬 회고
- **콘솔 프로젝트 &rightarrow; 스프링부트 프로젝트로 전환**
  - 콘솔 프로젝트 도중, Gradle 등 환경 설정 과정에서 계속 오류가 나서 불필요하게 너무 많은 시간을 소모했다.

- **엔티티 클래스의 기본 생성자**
  - JPA에서는 엔티티를 리플렉션(Reflection)으로 생성 &rightarrow; 엔티티 클래스에는 반드시 **public 또는 protected 접근 제어자를 가진 기본 생성자**가 필요
  - `@NoArgsConstructor(access = AccessLevel.PROTECTED)` &rightarrow; Lombok이 자동으로 protected 기본 생성자를 생성, 따로 작성할 필요 X

- **`@Autowired` (의존성 주입)**
  - `@Autowired`는 스프링 컨테이너가 관리하는 빈(Bean)을 자동으로 주입하는 애너테이션
  - 개발자가 직접 new 키워드로 객체를 생성하지 않아도,
스프링이 해당 객체를 생성하고 주입해줌
  - 객체의 생성과 관리를 스프링이 담당하므로,
클래스 간의 결합도가 낮아지고, 코드의 유연성과 유지보수성이 높아짐

- **DAO vs DTO**
  - DAO: 데이터베이스와 연결되어 데이터를 조회, 수정하는 데 사용하는 객체 (데이터 수정과 관련된 로직 포함)
  - DTO: 단순히 계층 간에 데이터를 옮기기 위해 사용하는 전달자 역할을 하는 객체 (별도의 비즈니스 로직 포함 X)
