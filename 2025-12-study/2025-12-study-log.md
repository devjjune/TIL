- ‘김영한의 실전 자바 - 기본편’ 중 4~6장의 연습 문제를 풀며 배운 점을 정리해보았다.      
  - **생성자 체이닝**: 가장 파라미터가 많은 생성자만 필드를 직접 초기화하고(책임 집중), 나머지 생성자는 `this(...)`로 그 생성자를 호출해 중복 코드를 제거한다. (오버로딩의 활용도 up)  
  - 조건문에서 **if-else 대신 `return;`으로** 메서드를 즉시 종료시키면, 들여쓰기를 하지 않아도 되고 핵심 로직을 더 잘 드러낼 수 있다. 
  - **지역 변수**는 '메서드 안에서 잠깐 쓰는 값'이라 개발자가 직접 초기화해야 하고, **필드 변수**는 '객체의 속성'이라 JVM이 자동으로 기본값을 넣어 안정성을 보장한다.
  - **속성과 기능은 가까울수록 좋다.** 즉, 관련된 데이터(필드)와 그 데이터를 다루는 기능(메서드)을 같은 클래스 안에 묶어야 객체지향적으로 좋은 설계가 된다.
  - 향상된 for문은 배열 전체를 처음부터 끝까지 순회하지만, 일반 for문은 `i < itemCount`처럼 **실제 채워진 범위만 순회**하도록 제어할 수 있다. (부분적으로만 데이터가 있는 배열에서는 일반 for문이 더 적절함)

# 🗓️ 2025-12-06 (토)
**1. 자바 코드 연습**
- ‘김영한의 실전 자바 - 기본편’ 중 7~9장의 연습 문제를 풀며 배운 점을 정리해보았다.  
  - getter은 **외부에서** private 필드 값에 직접 접근할 때만 필요하다. private 필드 값을 **클래스 내부에서만** 사용할 경우엔 getter가 없어도 된다.
  - private 메서드는 오직 같은 클래스 내부에서만 호출할 수 있다. (외부에 보여주지 않을 내부 로직을 숨기기 위해서) 다른 클래스에서 쓰고 싶으면 public 메서드로 감싸야 한다. 
  - **캐스팅(casting)**: 어떤 타입의 값을 다른 타입으로 강제로 바꾸는 문법 `(변환할 타입) 값`
  - `import static`: 클래스의 static 멤버(필드, 메서드)를 클래스 이름 없이 바로 사용할 수 있게 해주는 문법 (단축키 option + Enter). _단, 남발하면 출처 클래스가 불분명해짐._
  - **클래스를 상속받으면 부모의 필드와 메서드를 별도의 선언 없이 그대로 사용할 수 있다.** 단, 부모에 기본 생성자가 없으면 자식 생성자에서 `super(...)`를 호출해야 하고, 부모 메서드를 바꿔 쓰고 싶을 때만 `@Override`로 재정의하면 된다.
  - 부모 메서드를 오버라이드할 때, 접근 제어자를 ‘더 좁게’ 만들면 안 된다.

- 전체적인 리팩토링은 완료했고 프로그램도 성공적으로 실행됐다. 이후엔 접근 제어자를 수정하고, 예외 사항에 대한 검증 로직을 추가할 예정이다.

  - **model 객체를 새로 생성**해 각 객체가 책임질 메서드를 할당했다.
  - **바로 전 계산기 문제보다 service 로직이 복잡해졌다.** 이전 문제에서는 객체에서 모든 기능을 수행해 service에서 그냥 호출하기만 하면 됐었는데, 이번에는 각 객체에서 호출한 기능을 조합해서 service 흐름을 만들어야 했다. 
  - 또한 이전 계산기 문제에서처럼 출력 로직이 메인 로직과 분리되어 있는 것이 아니라, **메인 로직이 실행되는 동안에도 각 회차의 결과를 출력**해야 했다. 

- **AI의 도움을 받지 않고 구현**했다.

- **커밋 로그 가독성**에 대해 고민했다. 리팩터링 시 작은 수정은 교차 diff가 적합하지만, 파일 전체를 재구성하는 경우에는 변경 전·후를 두 덩어리로 분리하는 것이 나은 것 같다.
